# PSK Session Key Exchange

This document describes the pre-transfer key establishment added to the
one-FPGA dual-SoC loopback design in `aes_soc_top_dual_txbram_rxsd`.

## Goal

Before the image transfer starts, the TX and RX PicoRV32 systems now derive
fresh session keys from:

- one long-term pre-shared key (PSK)
- one nonce generated by RX
- one nonce generated by TX

The derived session keys are:

- `Kenc`: used for AES image encryption/decryption
- `Kmac`: used for AES-CMAC over the ciphertext stream

`K1`, the CMAC final-block subkey, is derived at runtime from `Kmac`.

## Why mailbox-based exchange was chosen

The current one-FPGA prototype has a unidirectional TX->RX SPI data path for
the payload stream. There is no reverse SPI link from RX back to TX in this
prototype.

For that reason, the existing mailbox is used as the control plane for the
key-establishment phase:

- RX publishes `nonce_rx`
- TX reads `nonce_rx`, generates `nonce_tx`, and publishes it
- RX reads `nonce_tx`
- both sides derive the same session keys locally

This keeps the data-plane unchanged while still proving session-key
establishment in simulation.

## Mailbox usage

The mailbox already exposed:

- `flags`
- `expected_blocks`
- `tx_count`
- `rx_count`
- `aux0`
- `aux1`

The handshake uses:

- `aux0` = `nonce_rx` during setup
- `aux1` = `nonce_tx` during setup
- `flags[5]` = `FLAG_KEYS_READY`

After the run, `aux0` is reused for the final image/MAC status bits as before.

## BRAM data layout additions

The generator now uses these extra BRAM regions:

- `PSK_BASE = 0x1CD0`
- `NONCE_RX_BASE = 0x1CE0`
- `NONCE_TX_BASE = 0x1CF0`

Existing regions are reused as runtime session outputs:

- `KEY_BASE = 0x1C60` stores derived `Kenc`
- `MAC_KEY_BASE = 0x1C70` stores derived `Kmac`
- `CMAC_K1_BASE = 0x1C80` stores runtime-derived `K1`

These values are stored in BRAM on both TX and RX so the testbench can inspect
and compare them directly.

## Handshake flow

### RX side

1. Wait for SD init and button press.
2. Generate `nonce_rx` from a simple software-evolved counter.
3. Store `nonce_rx` in RX BRAM and write it to mailbox `aux0`.
4. Clear mailbox `aux1`.
5. Set mailbox `FLAG_START`.
6. Wait for TX to publish `nonce_tx` in mailbox `aux1`.
7. Store `nonce_tx` in RX BRAM.
8. Derive `Kenc`.
9. Derive `Kmac`.
10. Derive `K1` from `Kmac`.
11. Write metadata and the derived `Kenc` sector to SD.
12. Initialize CMAC and the AES decrypt key.
13. Set mailbox `FLAG_KEYS_READY`.
14. Begin receiving ciphertext blocks.

### TX side

1. Wait for mailbox `FLAG_START`.
2. Generate `nonce_tx` from a simple software-evolved counter while polling.
3. Read `nonce_rx` from mailbox `aux0`.
4. Store both nonces in TX BRAM.
5. Write `nonce_tx` to mailbox `aux1`.
6. Derive `Kenc`.
7. Derive `Kmac`.
8. Derive `K1` from `Kmac`.
9. Wait for mailbox `FLAG_KEYS_READY`.
10. Initialize CMAC and begin the normal encrypted transfer.

## Session key derivation

The PSK is stored in BRAM at `PSK_BASE`.

Both TX and RX derive keys using the AES encrypt custom instruction with the
PSK loaded as the AES key.

Two derivation blocks are formed:

- block for `Kenc`: `nonce_rx || nonce_tx || "KENC" || "PSK1"`
- block for `Kmac`: `nonce_rx || nonce_tx || "KMAC" || "PSK1"`

Then:

- `Kenc = AES(PSK, derive_block_kenc)`
- `Kmac = AES(PSK, derive_block_kmac)`

These are written into `KEY_BASE` and `MAC_KEY_BASE`.

## Runtime derivation of CMAC K1

Because `Kmac` now changes every session, the CMAC subkey `K1` can no longer
be precomputed offline.

The firmware now derives it at runtime:

1. Compute `L = AES(Kmac, 0^128)`.
2. Store `L` into `CMAC_K1_BASE`.
3. Shift the 16-byte value left by one bit in software.
4. If the original MSB was `1`, XOR the last byte with `0x87`.

The resulting 16-byte value becomes `K1`.

## Transfer flow after handshake

Once the session keys are ready, the rest of the transfer works like before:

- TX encrypts image blocks with derived `Kenc`
- TX computes CMAC over the ciphertext stream with derived `Kmac`
- TX sends a final tag block
- RX decrypts with derived `Kenc`
- RX recomputes CMAC with derived `Kmac`
- RX compares the received tag against its locally computed tag

PASS requires:

- decrypted image matches expected image
- CMAC tag matches

## Simulation verification

The testbench `tb_dual_txbram_rxsd.v` now checks the handshake directly.

### Normal case

Expected:

- TX nonce matches RX nonce record
- TX `Kenc` matches RX `Kenc`
- TX `Kmac` matches RX `Kmac`
- TX `K1` matches RX `K1`
- image decrypts correctly
- final CMAC tag matches
- PASS is asserted

The testbench prints:

- TX/RX nonces
- TX/RX `Kenc`
- TX/RX `Kmac`
- TX/RX `K1`
- final TX/RX CMAC registers
- received tag and local tag

### Tampered tag case

Compile the testbench with:

- `TAMPER_RX_BLOCK_INDEX=196`

This flips one bit in the final received tag block.

Expected:

- derived session keys still match
- image still decrypts correctly
- tag mismatch is detected
- FAIL is asserted

### Bad PSK case

Compile the testbench with:

- `BAD_RX_PSK=1`

This flips one bit in the RX PSK BRAM contents before reset release.

Expected:

- TX/RX nonces still match
- TX/RX session keys diverge
- decrypt output is wrong
- tag comparison fails
- FAIL is asserted

## Limitations

- Nonces are software-generated from simple counters, not a cryptographic RNG.
- In the current one-FPGA prototype, nonce exchange uses the mailbox control
  plane rather than a bidirectional physical transport.
- The existing PC-side SD verifier is not yet extended to reconstruct and
  verify the session keys or CMAC from hardware runs.

Even with those limitations, this implementation proves the intended session
key-establishment behavior and shows that a PSK mismatch causes the transfer to
fail end-to-end.
